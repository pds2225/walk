# walk 사진 업로드 기능 — 개발계획안 (계획만, 미구현)

> 작성 2026-07-13 (사용자 요청: "계획안만 md+위키"). 구현 착수 전 **결정 필요 2가지**(§5) 확인 요망.

## 1. 무엇에 쓰는 사진인가 (용도 후보 — 택1 또는 조합)
| 후보 | 내용 | 가치 |
|---|---|---|
| **A. 도착 기록(추천)** | 도착 시 "📷 도착 사진 남기기" → 산책/방문 기록에 날짜·목적지와 함께 저장 | 산책 앱들의 검증된 기능, 구현 최소 |
| B. 장소 메모 | 즐겨찾기/목적지에 사진 첨부("이 입구로 들어감") | 재방문 시 유용, 저장 필요량↑ |
| C. 문제 지점 신고 | 걷다가 위험/공사 지점 사진+좌표 기록 | 개인용으론 활용도 낮음 |

## 2. 기술 방법 (Streamlit 내장, 신규 의존성 0)
- **촬영**: `st.camera_input()` — 폰 브라우저에서 카메라 바로 열림(권한 프롬프트).
- **앨범 선택**: `st.file_uploader(type=["jpg","jpeg","png"])`.
- 리사이즈: PIL(이미 requirements에 있음)로 긴 변 1280px·JPEG 80% 압축(용량 ~200KB).
- 표시: 기록 목록에 `st.image` 썸네일.

## 3. ⚠ 저장소 제약 (핵심 리스크 — 결정 필요)
Streamlit Cloud는 **파일시스템이 휘발성**(재배포·재시작 시 삭제). 서버에 저장하면 사라진다.
| 저장 방식 | 지속성 | 용량 | 비용/키 | 판정 |
|---|---|---|---|---|
| 세션 메모리 | 그 세션만 | 충분 | 없음 | MVP 1단계(임시 확인용) |
| **브라우저 localStorage(base64)** | 그 폰에 영구 | **총 ~5MB(사진 ~20장)** | 없음 | **MVP 추천** — 기존 즐겨찾기/최근검색과 같은 방식 |
| 클라우드 스토리지(S3/Supabase 등) | 영구·기기간 공유 | 사실상 무제한 | **키·계정 필요** | 2단계(사진 많아지면) |
| GitHub repo 커밋 | 영구 | 부적합(용량·공개) | — | 기각 |

## 4. 단계별 계획
1. **MVP**: 도착 화면에 `st.camera_input` → PIL 압축 → localStorage 저장(`walk_photos` 키, 기존 `_save_list_to_ls` 재사용) → "산책 기록" expander에 날짜·목적지·썸네일 목록 + 개별 삭제. 테스트: 압축 순수함수 단위테스트 + 스모크.
2. **확장(선택)**: 즐겨찾기별 사진 첨부(B), 용량 관리(오래된 것부터 자동 정리), 클라우드 저장 전환.
- 예상 규모: MVP = 파일 1곳(1_Navigation) + 압축 헬퍼, 라이브러리 추가 없음.

## 5. 착수 전 결정 필요 (사용자)
1. **용도**: A(도착 기록)로 시작해도 되는가? (B/C 병행 여부)
2. **저장**: localStorage(무료, 폰 한정 ~20장)로 충분한가, 처음부터 클라우드(키 필요)인가?

## 6. 리스크
- localStorage 5MB 초과 시 저장 실패 → 저장 전 용량 체크+오래된 사진 자동 정리 필수.
- iOS Safari의 camera_input 동작 편차 → 실기기 확인 항목.
- 사진은 개인 데이터 — 외부 전송 없음(로컬 저장만)을 기본 원칙으로.
