# walk 스냅투루트/맵매칭 — 진행도 기반 이탈 판정 (개발 참고 설계서)

> 저장일: 2026-07-12 · 상태: **설계·참고자료(구현 대기)** · 성격: **참고용, 절대 기준 아님**(사용자 지정)
> 목적: GPS 튐·정지로 인한 **헛 재탐색 제거** + 진짜 이탈은 감지. `/clear` 후에도 개발이 이 문서를 참조.
> 관련 근거: 이 세션 시뮬레이션(쿨다운 2~12초 = 재탐색 10회 동일 / 위치평균 10→9 / 맵매칭 코리도어 10→0).

## 0. 한 줄 결론
쿨다운·샘플수 튜닝은 **곁가지**(효과 없음). 근본 해결 = **경로선까지 횡거리 단독 판정 → "실제로 어느 방향의 어느 도로로 이동 중인가"(진행도+이동+방향) 기반 판정**으로 전환.
핵심 변경 3가지: ①쿨다운 3초 통일(완료 PR#46) ②**진행도 유지·감소를 직접 이탈 조건에서 제외** ③6~8m 명확 오진입은 조기 재탐색.

## 2. ⚠️ 데이터 제약 (반드시 인지)
앱이 아는 것 = **① 계획 경로 1개(polyline+turn_points) ② GPS**뿐. **주변 도로망 지도 없음.**
→ **평행도로 식별·다른 세그먼트 적합도·일반 교차로 인근 판정은 구현 불가**(도로/보행 네트워크 데이터 API를 새로 붙여야 함 = 대규모 확장).
→ 이번 범위: **단일 경로 + GPS + 이동(진행도/방향/속도)** 로 되는 것만. 그래도 사용자 목표(오탐 제거+진짜 이탈 감지)는 달성 가능.

### 되는 것 vs 안 되는 것
- ✅ 스냅투루트(계획 경로 투영), 진행도 기반 이탈, 정지/신호대기 보호, 순간 튐 무시, 실제이동+방향+지속 확정, 6~8m 조기 이탈, GPS정확도 동적 코리도어
- ❌ 평행도로 판별, 일반 교차로 인근 판정, 다른 골목 세그먼트 매칭 (도로망 데이터 필요)

## 3. 설계 (engine.py 코어 비침습)
- **신규 `snap_router.py`(순수함수·상태머신)** — 엔진은 손대지 않고, 엔진 metrics(횡거리·진행거리)+raw 이동을 받아 재탐색 여부를 확정.
- **상태머신**: ON_ROUTE / STATIONARY / UNCERTAIN / OFF_ROUTE_CANDIDATE / OFF_ROUTE_CONFIRMED / REROUTING / COOLDOWN. 상태별 **진입/유지/해제 조건 분리**(매 틱 토글 방지).
- **1_Navigation 배선**: 엔진 `deviated`=후보로만 → snap_router가 CONFIRMED일 때만 `_fetch_route`. **REROUTING 잠금**(쿨다운만으론 API 중복 못 막음 — isRerouting 별도).

### 상태 전이(요지)
| 상태 | 진입 | 해제 |
|---|---|---|
| STATIONARY | 일정 시간 이동 없음(이동거리<accuracy, 속도<0.5, 진행도 정체) | 유효 이동 3~5m |
| UNCERTAIN | GPS accuracy 불량(>30m) | 양호 샘플 2회 |
| OFF_ROUTE_CANDIDATE | 복수 이탈 근거(이동+횡거리↑+방향불일치) | 정상 적합도 회복 |
| OFF_ROUTE_CONFIRMED | 확정 조건 충족(연속 2~3 유효샘플) | 재탐색 시작 |
| REROUTING | API 요청 | 새 경로 적용/실패 |
| COOLDOWN | 새 경로 적용 완료 | 3초 |

## 4. 파라미터(초기 가정값 — 상수/설정 분리)
- 경로 허용거리 기본 15m / 동적 최대 30m(GPS accuracy·도심 반영) / **최대 스냅거리 상한**(실이탈자 고착 방지)
- accuracy>30m → 횡거리 조건 신뢰도 하향(단독 사용 금지)
- 정지: 최근 3~5초 이동<3m, 속도<0.5m/s, 이동<accuracy, 진행도 정체 → **속도 단독 금지**(느린 보행/노인/휠체어 오판 방지)
- 진행도 후퇴 허용 3~8m(accuracy 나쁘면 확대), 10m+ 후퇴는 **반대방향 실이동 확인 시에만** 보조근거
- 조기 이탈: **accuracy≤10m일 때만** 6~8m 오진입+방향차 60~90°+지속 → 1~3초 내 재탐색

## 5. 필수 테스트(시뮬 재현)
1. 정상경로 20~30m 지속편향 → 재탐색 0 2. 순간 40m 튐→복귀 → 0 3. 횡단보도 정지 흔들림 → STATIONARY, 0
4. 평행도로 실진입 → (제약: 근사만) 5. 반대방향 실이동 → 5~10초 내 재탐색 6. 교차로 정상회전 → 0
7. 교차로 오회전(6~8m) → 1~3초 조기 8. 경로위 정지(진행유지) → 0 9. accuracy30~50m → UNCERTAIN 보류 10. 재탐색 직후 흔들림 → 3초 중복 0
+ U자/왕복 경로에서 **지나온 세그먼트로 스냅되는 문제**(세그먼트 탐색범위 제한) 반드시 테스트.

## 6. KPI(완료 기준 — 재탐색 횟수만으론 판단 금지)
정상경로 오탐률 ≤1% / 실제이탈 탐지율 ≥95% / 탐지지연 평균 ≤3초 / 짧은골목 감지거리 ≤8m / 정지 오탐 0% 목표 / 교차로 정상회전 오탐 ≤3% / 중복 재탐색 ≤1% / 스냅 오류율 ≤3%.
**최종 기준**: (a) 정상·정지·튐에서 오탐 0 + (b) 실제 다른길/반대방향은 적정시간 내 재탐색. **둘 다** 충족.

## 7. 최상위 리스크(난도순)
1. **평행도로** — 폴리라인만으론 구분 불가(도로망 필요, 이번 범위 밖)
2. **짧은골목 6~8m 조기감지** — GPS 나쁘면 오탐(accuracy≤10m 게이트)
3. 스냅 과강→실이탈 놓침 / 스냅 과약→기존 오탐 유지 (코리도어 균형)
4. 정지 보호 과도→느린 실이탈 놓침
5. 쿨다운(대기)과 이탈확인시간 혼동 금지 — 쿨다운=재탐색 후 중복방지용
6. 교차로 세그먼트 오전환→진행도 전체 흔들림
7. 재탐색 중복방지=쿨다운 외 **REROUTING 잠금 필수**
8. U자/왕복=지나온 세그먼트 스냅(탐색범위 제한)

## 8. 개발 절차
단위테스트(최근접점·진행도·방향·코리도어·정지·쿨다운) → 시뮬(§5) → (현장 실보행은 사용자 몫: 홍대·연남·합정, 교차로·짧은골목·평행도로) → 회귀(마커·음성·분기안내·도착·재탐색·백그라운드 복귀).
성능: 직전 세그먼트 주변만 우선검색(전체 스캔 회피), 위치 1회 처리 ≤50ms 권장.

## 10. 모드 분리 아키텍처 (도보 vs 도보+대중교통 복합) — 참고, 절대기준 아님
**결론: 두 모드 분리. 단 앱은 하나. 공통 위치엔진 + 모드별 경로판정기.**

**왜 분리(같은 이탈판정기 쓰면 조건문 폭증·테스트 난망):**
| 상황 | 도보모드 | 복합모드 |
|---|---|---|
| 버스 탑승 | 이탈 | 정상 진행 |
| GPS 소실 | 예외/불확실 | 지하철 구간 정상 |
| 고속 이동 | 비정상(탈것 의심) | 버스·지하철 정상 |
| 진행도 | 경로 폴리라인 누적 | **구간별 진행도** |
| 도착 판정 | 최종 목적지 | 정류장·역·환승지·최종 |
| 재탐색 | 현재위치→도보 재생성 | 전체 여정/해당 구간 재계산 |

**구조:** Navigation Core(Location Engine·GPS Quality·Reroute Manager·음성·마커·오류) 공통 → Walking Engine / Multimodal Engine 분리. `NavigationMode`={WALK_ONLY, MULTIMODAL}, `RouteSegment.segmentType`={WALK,BUS,SUBWAY,TRANSFER,FINAL_WALK} → **도보 엔진을 복합의 도보 구간에서 재사용**.

**상태머신:**
- 도보: WALKING/STATIONARY/UNCERTAIN/OFF_ROUTE_CANDIDATE/CONFIRMED/REROUTING/COOLDOWN/ARRIVED + (도보중 탈것) **NON_WALKING_CANDIDATE/CONFIRMED**(→도보 이탈판정 일시중지+모드전환 제안).
- 복합: WALK_TO_STOP/WAITING_FOR_TRANSIT/BOARDING_CANDIDATE/IN_BUS/IN_SUBWAY/ALIGHTING_CANDIDATE/TRANSFER_WALK/FINAL_WALK/REROUTING/ARRIVED. 전체 진행도보다 **구간 인덱스+구간별 진행도**가 핵심.

**자동감지 원칙(중요):** 자동감지는 **모드 강제전환용이 아니라 "잘못된 재탐색을 막는" 보호용.** 모드 전환은 **사용자 확인 우선**(달리기→버스 오인, 택시→노선 오매칭, 역주변 GPS소실→지하철 오인 방지). UX: "보행이 아닌 이동 감지 → [대중교통 경로 보기]/[도보 계속]", 확인 전까지 도보 이탈판정 일시중지.

**개발 우선순위:** ①도보모드 안정화(=snap_router, 현재 범위) → ②비보행 상태 보호(도보중 고속→도보 재탐색 일시중지+지하철역 GPS소실 보호+전환 안내) → ③복합경로모드 별도.

**MVP 범위:** 도보 스냅투루트 안정화 + 비보행 이동 감지 + 대중교통 탑승 추정 시 재탐색 일시중지 + 복합 전환 안내. **복합경로모드 자체는 후속 버전.**

**현 코드 연결점:** 이미 `transit_builder.py`(Journey/JourneyLeg WALK/subway/bus, `advance_leg`, 도보 leg는 도보경로 재사용)가 **복합모드 경로 모델의 씨앗**. 단 **탑승 감지·비보행 보호·구간 재탐색은 미구현**. 즉 복합모드는 0에서 시작 아님(모델 존재)이나, 이탈/재탐색 판정은 신규.

**난이도:** 복합모드 ≈ 도보모드의 **2~3배**(외부 노선·정류장·실시간 데이터, 상태 10~15개, 시나리오 50~100개). → 도보 먼저.
