# walk 소셜 비전 — 걸은 길 경험 공유 (문제인식 / 해결방안)

> 2026-07-13 사용자 선언(중요정보). 위키 `walk-social-vision.md`·메모리 연동. 구현 전 방향 문서.

## 1. 문제인식 (Problem)

1. **도보 길안내는 '선과 화살표'뿐이라 초행자는 여전히 헤맨다.**
   지도앱은 경로선·회전 지시를 주지만, 실제 보행자가 막히는 지점은 **"어느 골목 입구인지, 어떤 건물 옆인지"** 같은 현장 맥락이다. 이 프로젝트의 실기기 테스트에서도 GPS 오차·회전 놓침이 반복 확인됐다 — 좌표 정보만으로는 마지막 순간의 확신을 못 준다.

2. **기존 지도앱의 공유 기능은 있으나 쓰이지 않는다.**
   네이버지도는 장소 저장·그룹 공유를 제공하지만 실사용이 거의 없다(사용자 관찰). 원인: 공유 단위가 **'장소 리스트'**라 정보만 있고 **경험이 전달되지 않는다** — 왜 좋은지, 어떻게 가는지, 그 길이 어떤 느낌인지가 빠져 있다.

3. **걷기 기록 앱은 운동에 치중, 길찾기와 분리되어 있다.**
   산책·러닝 앱의 기록(궤적·거리)은 운동 통계로 끝나고, 그 경험이 **다음에 같은 길을 찾는 사람을 돕는 데 재사용되지 않는다.** 경험과 길안내가 단절된 상태.

## 2. 해결방안 (Solution)

**"가는 길 자체"를 사진과 함께 경험 단위로 기록·공유하는 소셜 네트워크를 만들고, 그 경험이 길찾는 사람의 내비게이션을 직접 돕게 한다.**

1. **경험 단위 기록**: 안내 종료(도착) 시 걸은 궤적(nav_samples) + 거리·시간(도착 요약) + **현장 사진**(갈림길·입구·풍경) + 한 줄 메모를 하나의 '산책 기록'으로 저장.
2. **공유**: 기록을 링크/이미지로 공유(1차: 계정 불필요) → 팔로우·피드(2차: 백엔드 도입 후)로 **걸은 경험의 소셜 네트워크** 형성.
3. **길찾기 선순환(핵심 차별점)**: 다른 사람이 같은 목적지로 길을 찾을 때, **실제로 걸은 사람의 사진과 경로**를 안내에 결합해 보여준다 — "이 골목 입구(사진)에서 좌회전". 회전 지점·입구처럼 헤매기 쉬운 곳의 실사 컷이 좌표 안내의 빈틈을 메운다.
   → **기록이 쌓일수록 길안내가 좋아지는 구조**: 경험 공유 = 콘텐츠이자 내비 데이터. (네이버 장소공유와 달리, 공유물이 소비되는 이유가 명확)

## 3. 단계별 실행 (기존 자산 재사용)

| 단계 | 내용 | 필요한 것 |
|---|---|---|
| **1. 개인 산책 기록 (MVP)** | 도착 시 궤적+사진+요약을 localStorage 저장, "산책 기록" 목록·상세(궤적 지도 재생) | 지금 스택으로 가능 — nav_samples·도착요약·`_save_list_to_ls`·사진 계획안([[walk-photo-upload]]) 재사용 |
| 2. 공유 링크/이미지 | 기록을 정적 지도+사진 카드로 내보내 카톡 등 공유 | 지금 스택으로 가능 |
| 3. 소셜 네트워크 + 내비 연동 | 계정·피드·팔로우, 목적지 검색 시 타인의 경험 사진을 안내에 표시 | **외부 백엔드(DB·스토리지·계정) 필요 — 큰 아키텍처 결정.** 1·2 사용성 검증 후 |

## 4. 제약 (정직)
- Streamlit Cloud는 서버 저장이 휘발성 → 1·2단계는 폰 로컬(localStorage ~5MB) 한계 안에서, 3단계는 Supabase 등 백엔드 도입이 전제.
- 사진·궤적은 민감한 개인 위치 데이터 — 공유는 **항상 사용자가 명시적으로 선택**(기본 비공개).
